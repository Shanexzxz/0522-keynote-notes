上午 Keynote 演讲纪要

1
别让 AI Coding "看起来很美"——从评测到落地，填坑与新架构实践之路
茹炳晟· 复旦大学 CodeWisdom 首席技术专家 / 腾讯研究院特约研究员· 09:00–09:40
▼

一、AI 在企业能否有效落地，核心是管理水平，不是模型问题

  • 把模型该有的上下文给到位，模型一定会做。比如让模型理解你的业务、判断你的文档是否支持某个功能——把高版本和低版本两个文档一起扔给模型，再让它确认是否支持，模型是能给出结论的。
  • 在企业里能不能有效落地，核心不是模型问题，而是企业的管理水平。管理才是根本。
  • 对大型传统软件而言，AI 也带来了很多新机会：可以通过 Low-Code 快速构建原型，让需求的探索和启发成本变低。
  • 由此带来研发模式的变化：
    • 不再是需求都没确定就让开发实现，实现完再来回改，造成研发资源巨大浪费；
    • 而是产品经理通过原型反复体验来确定最终产品形态——需求的启发和探索由产品经理 + Coding 同学自闭环，需求明确之后再投入正式研发流程；
    • 这样有限的研发资源就可以得到更高效的利用。

二、SD（Software Design）模式在传统软件下的有效边界

  • Verbal（口头描述）肯定不靠谱。
  • SD 模式是否靠谱，要看场景：
    • 靠谱：新功能或代码改动不依赖 / 很少依赖原有代码。比如新加一个功能在新的位置上实现，原来通过 API 接口去访问——这种情况 SD 依然有用。
    • 不靠谱：绝大多数传统软件代码像意大利面条一样揉在一起，一个改动往往牵涉多处，人都要跑一圈才能找全。这种情况下让 AI 用 SD 模式硬改，基本上是推卸责任。
  • 这种模式理论上很优美，落地的时候会举步维艰，根本做不下去——局限性会被放大。

唯一判断标准：对"隐性知识"的依赖度

  • 在传统软件体系下，AI / AI 产品 / Workflow 能不能有效，唯一判断标准就是：做这件事本身需不需要大量的隐性知识。
  • 如果对隐性知识依赖很高，而这些隐性知识没有被收进去（还停留在平时会议里的口头交流），那么这个任务一定做不好。
  • 如果对隐性知识依赖很少，或者隐性知识已经被知识工程显性化出来，那么这类业务一定可以做。

三、即兴软件：AI 时代的新软件形态

定义与特征

  • 传统软件是开发完固定在那里的；即兴软件是因为 AI 引擎现在很强，用户直接提需求 → AI 直接转换成可运行代码 → 通过可运行代码产生结果。代码本身变成一个不重要的过程性手段。
  • 在传统软件开发里，提示词的价值是 "Prompt to Code"——通过提示词转换成代码；而在即兴软件下，Prompt 本身就是 App。

与微信小程序的区别

  • 微信小程序也叫"用完即走"，但它本身是用传统方式预先开发好的，只是运行态用完即走。
  • 即兴软件是开发态和运行态都用完即走：当场开发、当场上线、当场发布。
  • 这是一个完全的新品类，是 Vibe Coding 的主战场。

满足原本未被满足的增量市场

  • 例如：今天这个会议想做个抽奖程序，这种软件以前没人帮你做，现在用 Vibe Coding 可以快速搞出来。原本没有被满足的市场，今天可以被快速满足，这种长尾需求其实是茫茫多的。
  • 不要看不起这个市场。专业人员一开始会判断这玩意儿就是个玩具、就是个 demo。
  • 但即兴软件的发展路径会走过三个阶段：看不上 → 看不起 → 看不懂。

颠覆式创新的三大特点 ✅ 即兴软件全部命中

  • 成本低
  • 规模大
  • 足够个性化、门槛足够低

短视频开始也是看不上、看不起，到最后就看不懂了。即兴软件很有可能打出一个全新的市场空间，原本生态上没有这个生态位的公司会占据一个非常大的生态位。

四、即兴软件下 AI Coding 的核心能力

  • 一句话：凡是可以被验证的，就可以被实现。
  • 用自然语言把需求讲清楚的能力，在这个阶段变得非常重要。Prompt as Code——你的提示词能不能讲清楚，会成为一种非常稀缺的生产力。
  • 这对人提出了全新挑战，但不是替代传统的原生开发者，而是面向一类即兴软件的创造者，类似短视频早期的玩家。
  • 不是说不要原生开发了，而是要构建一套用大语言模型动态创建、动态推理、动态运行的全新运行态。这样的开发者平台未来可能成长为行业独角兽。
  • 未来形态想象：你跟平台说一个需求，比如"把图片转成图标"，它直接帮你生成一段代码，拿你的图片转成图标，用完即走。这样的平台可以成为个人助手非常好的入口。

五、即兴软件做大之后：重写还是重构？

  • 即兴软件一旦被做大做长，同样会陷入复杂性、架构很乱、后期没有掌控感。
  • 建议是不做重构，重写就可以了：
    • 用另一套体系，用标准化的 DDD（领域驱动设计） 去做设计；
    • 用标准化的软件工程能力去重新实现；
    • 而不是在那一坨由 refactoring 造成的东西上重瀑布——ROI 低，最后产出的质量也无法达到原始需求。

WorkFlow vs SE：怎么选？

  • 取决于你的目的：
    • 如果只是单一目的的个人自用，怎么用都行，不用考虑太多。
    • 如果是创业团队，原型以后想做大做强，那就要对代码有掌控感——这种时候选 SE。

SE 模式最大的坑

  • SE 流程是：需求 → Design → Pass → 代码（SDLC）。
  • 最大的坑是：工程师发现代码不符合预期时，忍不住直接修改代码。
  • 直接改代码之后会怎样？
    • 前面的 Pass 里就丢了这个改动；
    • 设计文档与代码不一致；
    • 以后维护时加载之前的 Design / Pass，发现跟代码不一致，整个体系就乱了。
  • 正确做法：克制冲动，回到 Pass 层去改 Pass，让 Pass 重新生成代码。
  • 类比：写高级语言生成汇编，发现汇编不达标，不能直接改汇编，一定要去改高级语言。

六、AI 原生时代的新三层架构

旧三层

前端（交互）/ 后端（逻辑）/ 数据库（存储）

新三层

  • 顶层 · 意图层：大模型、提示词工程、上下文工程的事情。
  • 中层 · 编排层：模型能力 + Tools，形成标准化的 Function Calling / Plugin Execution / ReAct 等组合。
  • 底层 · 执行层：确定性的执行。
  • 三层的执行过程都会有一套完整的知识库做支持。

执行层的关键问题：是确定性执行，还是依然让模型来跑？

  • 有人会把执行层做成几十步的模型单次调用，每一步都依赖模型，看上去 AI 原生度很高。
  • 但这种做法会带来长程任务的严重衰减：
    • 假设拆成 20 步，每一步都是模型调用；
    • 单次调用准确率 99%；
    • 整体准确率 = 0.99 的 20 次方，衰减非常厉害。
  • 解决方案：执行层应该尽量少用非确定性的模型调用，多用确定性的代码来保证执行效率和结果的可靠性。

确定性代码从哪里来？

  • 依然是传统软件工程师的领域，甚至可以是对传统软件的解构再重构。
  • 例如 ERP：把 ERP 的能力拆成一个个原子能力——可以是 Skill，也可以是确定性的 SC，哪怕是工具包都行。
  • 这些原子能力通过结构化表达（在什么场景下能实现什么能力）暴露出来。
  • 上层就可以根据用户需求动态编排成新的应用流程——编排可以用传统编排语言，调用确定性执行层。
  • 编排出来的 Plan 也可以通过结构化的方式表达和结构化验证，形成一种更高阶的智能。
  • 这意味着大量传统的 ERP、SaaS 软件可能面临结构上的重构。比如一个大型旅游网站，可能拆成很多原子能力，每个原子能力确定性执行；面对客户的不同需求，动态编排一个流程出来去调确定性执行层。

知识层 / 本体论 / 生成式 UI

  • 知识这一层讲得很泛，但 PPT 低配化模式将会成为一个非常关键的入口。
  • 本体论（Ontology）：对整个领域建模最好的一种结构化方法，本身就是结构化的。
  • 生成式 UI / 交互：每次流程编排是动态的，界面就不需要固定。基于当场确定的用户需求，当场生成需要的界面，去完成用户交互即可。

即兴软件如果做成比较大型的系统，同样会陷入复杂性、架构混乱、后期失去掌控感。这时要做重构吗？我们的建议是不做重构，重写就可以了——用另一套体系、用标准化的 DDD（领域驱动设计）去做设计，用标准化的软件工程能力去重新实现，而不是在那一坨 refactoring 的东西上重瀑布。因为 ROI 低，最后产出的质量也无法达到原始需求。

另外，即兴软件应该用 Workflow 还是 SE 方式？取决于你的目的：单一目的的个人自用怎么用都行；但如果你是创业团队、原型以后想做大做强——哪怕这个小锤子以后想做大做强——那就要对代码有掌控感，这种时候选 SE。

SE 模式有很多坑。SE 流程是：需求 → Design → Pass → 代码（SDLC）。最大的坑是：工程师发现代码不符合预期时，忍不住直接修改代码。 这种做法不允许——一旦直接改代码，前面 Pass 里就丢了这个改动，设计文档跟代码就不一致了；以后维护时加载之前的 Design / Pass，发现跟代码对不上，整个体系就乱了。

正确做法是克制冲动，回到 Pass 层去改 Pass，让 Pass 重新生成代码。这就好比写高级语言生成汇编，发现汇编不达标——直接改汇编是不可能的，一定要去改高级语言。

AI 原生时代的新三层架构

AI 原生软件本质上是什么？以前讲的旧三层是前端、后端、数据库（交互、逻辑、存储）。在 AI 原生时代，我们会迎来新三层：

  • 顶层 · 意图层：大模型、提示词工程、上下文工程的事情。
  • 中层 · 编排层：模型能力 + Tools，组合成 Function Calling、Plugin Execution、ReAct 等标准化形式。
  • 底层 · 执行层：确定性的执行。

这三层的执行过程都会有一套完整的知识库做支持。

关键问题来了：执行层是确定性执行，还是依然让模型来跑？很多人会把执行层做成几十步的模型单次调用、生成动态 Workflow，每一步又是模型执行。这种做法 AI 原生度看上去很高，但会带来长程任务的严重衰减——假设拆成 20 步、每一步都是模型调用，单次准确率即便高达 99%，整体准确率也只有 0.99 的 20 次方，衰减非常厉害。

所以执行层应该尽量少用非确定性的模型调用，多用确定性的代码来保证执行效率和结果的可靠性。确定性的代码从哪来？依然是传统软件工程师的领域，甚至可以是对传统软件的解构再重构。比如 ERP——把 ERP 的能力拆成一个个原子能力，可以是 Skill，也可以是确定性的 SC，哪怕是工具包都行。这些原子能力通过结构化表达（在什么场景下能实现什么能力）暴露出来，上层就可以根据用户需求动态编排成新的应用流程，去调确定性执行层。编排出来的 Plan 也可以通过结构化方式表达和验证，形成一种更高阶的智能。

这意味着大量传统的 ERP、SaaS 软件可能面临结构上的重构。比如一个大型旅游网站，可能拆成很多原子能力、每个原子能力确定性执行；面对客户的不同需求，动态编排一个流程出来去调确定性执行层。

知识层这块讲得很泛，但 PPT 低配化模式将会成为一个非常关键的入口。本体论（Ontology）是对整个领域建模最好的一种结构化方法，它本身就是结构化的。我们也会有生成式的 UI / 交互——每次流程编排是动态的，界面就不需要固定，基于当场确定的用户需求，当场生成需要的界面去完成用户交互即可。

  • Every PR I write: specification is written together with AI, the test is written together with AI, the review is happening together with AI. These three fundamental steps — specification, test, review — are all augmented by AI.

I don't believe genuinely new things are happening; we just have new tools we can leverage to support all the existing software-engineering best practices.

Q4 — Agentic patterns vs traditional (GoF) design patterns?

The GoF book is one of the most important books ever written in software engineering. But that book is more descriptive of the deterministic side of software engineering. Agents and LLMs add a non-deterministic side — multiple probabilistic agents instead of one single deterministic component. The two are complementary. I would strongly recommend considering the GoF book every time you want to write high-quality code — and consider my book too. The combination of the two is the best use.

When we talk about software engineering, implementation is becoming a smaller and smaller part of it.

There are four phases: specification, implementation, testing, deployment. Implementation and deployment are becoming easier — Kubernetes and other tools help. But spec and testing are where our contribution as humans is crucial. We are in the position to control the quality.

Future of testing? Testing is becoming the most important profession right now.

AI Coding 产品演进的趋势

随着 AI 能力不断提升，从大约 23 年我们灵码开始做补全。当时主要在编辑区做同行后续位置代码的预测、接续行的代码补全，或者跨文件的 Next Action Prediction。后面随着模型能力升级（特别是 Sonnet 3.5 这样的模型），我们发现结合 AI 可以做多文件自动化编辑，能够大幅度提效。

再往后，更高级的模型解锁后，我们发现给 Coder 或灵码一个比较复杂的任务，用纯自然语言描述，它也能够自己理解用户想做什么，然后结合代码库里的存量代码去检索、发现，去完成一个比较复杂的任务。

从纯 Coding Work 来看，我们的程序员朋友每天的工作时间分布——真正写代码的环节其实并不多，可能很多朋友的真正编程时间少于 50%。我们有很多时间花在跟产品经理对焦需求、设计技术方案上。Coding 这个环节目前已经被大幅度提效了。 但 Coding 完了之后还有很长的后续链路：深度 Review、修复、验证、测试、CI/CD、部署和发布。

我们也发现：结合 AI、结合 Agent，已经不只是完成一个任务、一个需求，而是完成一个端到端的解决方案的交付。 使用 Cling Agent 完成任务的用户规模在快速扩大，每个任务的复杂度、执行时长、模型交互轮次也在快速提升，token 呈现指数型爆炸式增长。

AI 平台三阶段演进

第一阶段（23–24 年）：股权阶段。当时我们 Code 团队和灵码团队同步投入到 AI Coding 行业，国内灵码会做得早一点。AI 更多是纯辅助角色，人的视线还停留在 IDE 编辑区，自己在主导。

第二阶段（24 年底–25 年初）：随着模型能力升级，Cursor、灵码这种多文件编辑能力出现——@ 几个文件，AI 就能结合任务描述完成编辑。这个阶段效率提升非常大，而且人的关注焦点开始迁移——从盯着编辑区，逐步往与 AI 对话的 Chat 区域迁移。

第三阶段（25 年底至今）：我们已经可以把很多任务委派、异步地让 AI 执行。人在过程中的主要环节是最开始和后面，中间的 Coding 阶段焦点已经挪出去了。相应的产品变化：用户关注的中间区域已经不是代码编辑区，变成了对话流或任务执行过程；用户关心的是结构化的产出、任务执行进度、验证结果。前两天发布的 Net Graph 直接把 IDE 市值干掉了。

从 Prompt 到 Context 再到 RAS

最开始的股权或 AI Chat 时代，更多是程序员怎么样描述好需求、设计精巧完美的 Prompt——Prompt 的质量对结果输出质量有非常大的影响。

再往后逐步进入以 Context 为核心的阶段。AI Agent 需要结合 Coding 工具，动态自主地发现仓库里跟用户任务描述相关的状态。这个阶段上下文是非常核心的。 另外，Agent 是长周期的多轮迭代，不是简单的一问一答——怎么做好长周期多轮迭代带来的上下文累积，以及超出 Context Window 后的上下文管理，在这个阶段非常重要。

到 25 年底 26 年初，国外提出 RAS 的概念非常火。在以 RAS 为核心的阶段，重要的不再只是写好需求或做好上下文文档管理，更多是让 AI 在我们的代码库或环境里更有约束、更有保障地完整运行起来。这里涉及多 Agent 架构、专业化垂直场景优化、约束系统设置——保障智能体不会因为犯错或边界 case 损坏电脑、删除命令。另外通过把验证和评审做重做厚，让任务完成的难度和周期更长。

Code 平台与定位

Code 的前身是灵码团队。我们去年八月份在全球发布了 Code。它定位为一款面向真实软件、面向真实程序员的产品——不是 Live Coding，而是 Agentic 编程平台，解决程序员真实研发需求。

平台分为全球版（效果优先，对接海外 SOTA 模型）和国内版（安全可信，对接国内最好的模型）。底层是统一的 Agentic Coding Platform，包含多 Agent 框架、增强的上下文机制、Tools 设计、环境管理。再往下是 Auto Scheduling——结合用户任务复杂度和仓库复杂度，动态智能选择最适合的模型。Auto Scheduler 让用户不用花大量时间研究 GPT 5.5、OPT 4.7、4.6 各自适合什么场景，自动达到比较好的性价比。

企业落地挑战

从 25 年开始，企业客户大范围落地 AI Coding 产品。挑战主要有：

  • 效果不达用户预期（复杂工程下尤为明显）；
  • 代码坏味比较严重；
  • 企业复杂工程效果有偏差——核心是知识管理的缺失。

知识管理也是 Code 在发布时就重点投入的领域。海外的卡帕西讲怎么样建立 Weakness。我们 Code 在 25 年 8 月份也已经做了相关投入，做了一个产品能力管理仓库知识。知识包含很多方面：代码反推出的架构知识、程序员日常对话中的知识——程序员经常纠正 AI、约束 AI，这个过程中 AI 也应该学习到知识，建立长期记忆。或者一些 Skills 还需要沉淀（当时还没有 Skills 这个概念）。

底层有四个基础保障——首先是安全与合规：不只是传输过程的安全，也包含本地代码、文件、数据的安全性。这方面我们有非常扎实的投入。

工程感知与知识沉淀

从工程感知角度看仓库的理解和知识沉淀。上层是感知层，做了几套底层能力支持 Agent 更好地理解仓库：

  • 向量检索
  • 代码知识图谱
  • 历史 BigCommit 信息融合——AI 时代代码产出非常多，但真实 commit 到仓库的代码质量较高、经过人工确认。

有些产品会广泛 grep，找出一大堆相关内容再疯狂 read，再拼接出认为相关的上下文——真正开始任务前，上下文已经累积得非常长。 在我们的场景下：你这个需求，我发现之前一些 commit 跟它有关系，那么当时 commit 改了什么代码，可以快速定位。代码知识图谱也能找到关联实体——比如要改订单某个能力，我能发现它会影响周边的用户、其他服务。通过这套机制，能更快定位代码，降低模型调用次数、上下文累积长度和最终成本。

底层沉淀几套组件：Retrieval 能力、记忆系统（5 月份发布升级，企业客户可以在企业内的不同开发成员之间做知识共享）、Semantic Search（区别于传统 RAG，结合动态搜索能力补充上下文）。

Repo Wiki

这个功能上线之初就有用户体感很好。传统上程序员被迫维护很多文档：新人进来想了解仓库会比较困难，前辈的介绍文档现写很累；代码演进快，维护文档成本很高。

我们的 Wiki 根据代码动态实时编写——把仓库架构、数据流、实体关系全部处理出来，帮助你快速理解仓库；代码修改时也会自动变更。它当时设计是给人看，但我们发现梳理出的知识在 Agent 干活时也可以使用——不只是给人看，也可以给 AI、给 Agent 看，能够有效提高 Agent 的检索效率。

Stack 驱动开发

这在去年到今年是个比较火的趋势，已经有大量程序员基于这套工作流完成复杂需求的端到端交付。

Stack 写出来的流程：理解用户需求 → 看与代码库已有代码的关系（完全新增？已有模块改进？多模块关联修改？）→ 比较新的知识结合外部互联网 retrieval（比如说我刚发布了 Code Agent SDK，要基于它开发功能，但这种新知识在模型里还没有）→ 写出完善 Stack。

比较好的 Stack 五大特征：

  1. 机器可读、易读（Markdown 比较好，最近也有讨论说 HTML 现在的模型也能比较好理解，只是会多消耗 token）；
  2. 没有歧义；
  3. 可追溯；
  4. 可验证；
  5. 未来可演进。

Code 发布第一天就内置了 Stack 能力，当时叫 Test Mode——Pass 的意思是完成任务。5 月 15 号大版本发布，把 Pass Mode 完全独立化为单独的视窗，可以在里面完成偏异步、偏稳态的长时端到端任务交付。

Multi-Agent 思考

对知识的思考沉淀出 Community 长期记忆、知识沉淀、知识卡片；对 Spec 的思考沉淀出 Question Mode；这条 GT 是我们对 Multi-Agent 的思考。

Multi-Agent 到底要不要？还是单 Agent 就能完成端到端任务？这在行业里、包括 top 公司都有非常多讨论。到 26 年 4–5 月份，几乎成为行业共识——一定需要多 Agent 架构去完成端到端任务。

今年三月份发布专家团模式：基于对软件工程的理解，内部定义了行业专家——编程专家、测试专家、Code Review 专家、Browser + Code Reviews 端到端验证专家。通过 Leader Agent 把专家串起来——用户给任务，Leader 思考要先搞清楚任务、列出实现计划、做端到端拆解：先 research、对仓库做了解，再编程；编程分前后端、分模块；模块间有依赖也可能没有依赖。没有依赖的模块完全可以并行——这在单 Agent 时期难以达成，多 Agent 时期可以做到并行化。

Agent 完成后比如后端做完发现某个模块不对——我们设计了一套收件箱机制让主子 Agent 通信。子发现问题反馈给主，由主找程序员澄清；主因为有相对全局视角，能看到前端有些东西已做完、不 block。Leader Agent 能做协调，比较好地完成任务。

另外做了端到端任务完成后的知识沉淀、Skill 沉淀，希望能做到越用越强。从原来只生成代码，到今天加入 Code Review 验证环节后，端到端交付一个结果。

5 月 15 号发布也是因为很多客户有需求——他们觉得有自己行业的专家，比如怎么定义后端代码、前端代码的写法、嵌入式领域的测试与我们定义的不一样。我们开放了专家自定义能力——完全新增领域专家，或在现有专家中融合行业 Skills、NCP，把行业 Know-how 融合到专家团里。

AI 驱动的前端自验证

世界上有 AI 软件能做相关事情，我们做了比较深入的投入——把前端验证能力完全内置到 Code 的 IDE 里。整个浏览器交互工具体系（包括主子 Agent 架构、智能模型路由）在浏览器 Agent 里完全实现。对比开源/社区方案，在 token 消耗和任务成本上有比较大优势。最后用户使用 Code 结合 Browser 验证，体验跟在 IDE 里使用终端验证差不多，不需要跳出 IDE。

前面发布已经上线云端 Use 功能。Browser Use 更多是 Web 开发——浏览器打开网页点点点、输入验证、登录报错。Computer Use 在浏览器之外拓展了验证能力——比如做 App 开发，今天要打开钉钉测试某功能正不正常，可以让 Computer Use 拉起钉钉点东西。也可以让它帮你配置不会用的软件——通过视觉能力看当前是什么样、应该点哪里。目前 Mac 下可用，我们也在探索 Linux 和 Windows。

多平台沙箱与 AI 驱动的智能拦截

最近半年 AI 在安全领域投入越来越大。模型输出导致的删盘、删数据 case 在收敛，但风险仍存在，一旦发生危害严重。

比如这里一个 case，他套了一个危险命令在嵌套命令里。传统 Coding Agent 可能认为执行的是 who am i 没问题，但没做多层命令解析就不知道里面套了非常危险的命令。所以第一层防护是我们做了一套完善的命令解析，适配 Windows、Linux、Mac 各种终端，提取出真实要执行的命令——嵌套或管道想绕都能精确提取。

提取后经过三层防护：通过模型判断命令有没有风险——这个例子里模型一般能识别出套了 rm 是非常危险的，会要求提权、使用沙箱执行；另外有黑名单和用户自定义黑名单——比如 rm 类命令，用户可能直接加入黑名单，默认走沙箱。

沙箱实现方式：给开发者完全隔离的本地 Docker 当然能解决文件安全问题，但完全隔离 Docker 在本地拉新的起来后，很多代码依赖、配置无法把本地工程很好地运行起来。所以我们没采取这种方案。 还有方案是别名替换——预定义几个危险命令（如 rm），发现包含 rm 就替换为放垃圾桶。但防护能力有限，只能防特定命令，比如通过 shell 脚本最后执行的是 shell 而不是 rm，就识别不出来了。

我们今天做的是轻量级本地化的、不是完全隔离的——只是在限定目录下限定 bash 执行权限。 macOS 下通过 CPL 实现：不完全约束权限，把当前用户打开的代码工程的读写权限完全开放给 shell——在工程范围内可以读写、删东西；但跳出当前工程（删其他目录、其他盘、整盘）的命令会被完全拦掉。macOS 下通过 SecCloud 实现；Linux 下用 Bubblewrap 做 Linux Namespace 级隔离；Windows 下用受信令牌的方式，通过 ACL 机制拦截命令权限。

内部实践案例一：用户工单全自动处理

第一个是用户反馈上报问题的全链路全自动处理。去年八月份上线之初就有一键上报按钮，把当前 IE 日志、截图传给我们告诉遇到了什么问题。从上线到现在，团队规模没扩大多少，但工单数量扩大了几十倍——处理这么多爆炸式工单非常人力不足。

传统模式：开发看工单 → 不同团队的人都要看 → 看跟我相关的 → 大概定位 → 复杂的下载日志逐行分析 → 不熟悉别人代码还要结合代码看 → 发现是系统问题就修复——这个流程非常长。

有了自动处理系统，底层基于 Code CUI 能力。上层收到反馈做智能分类——网络类、模型效果、Agent Loop 设计、稳定性、本地网络/代码问题。然后智能语义聚类——某类问题在半小时一小时内快速爆发会聚类预警，让研发快速接入。然后原生根因分析——结合用户问题描述、截图、日志，再把本地 Code 代码给到 CUI 分析。如果是明确的问题（error 或 exception），触发自动修复流程：自动识别用户的 IE 版本 → 切代码和分支 → 修复。如果是 UI 相关，结合 Use 用例去点点看；整个过程会做录屏，AI 完全自己做。最后修复完成在内部处理流程里完整可见——他在分析、改代码、录屏、验证。今天钉钉里时不时收到 PR——他说我发现用户感知问题，已完成修复，验证过程怎么样，给端到端的反馈。

效果：原先看一个工单要 30 分钟，现在一眼看大概知道是用户的问题、代码问题、底层 Infra 问题还是模型效果问题。整体效率提升超过 10 倍。40% 需求可以完成全自动端到端修复。 因为有海外用户 7×24 不停反馈，这套系统也是 7×24 不停在背后运转、分析、修复、提交。

内部实践案例二：Code Work

第二个是今年初发布的 Code Work。投入开发时收到一个挑战性目标——快速颠覆传统研发流程，交付一个完整的生产级 App。整个过程基本完全用 Code 开发，人几乎不真实触碰代码。

Code Work 是 React Native App，里面有大量前端代码。但开发过程中真实的前端工程师就一个左右，而且不是真实开发业务前端代码——他更多设计前端架构、整体组件交互、底层组件选型。业务层代码由好几个后端同学完成开发；前端 UI 部分很多由设计师同学完成开发——功能融合非常彻底。

整个过程完全用 Sketch 驱动：产品经理、程序员、设计师一起完善 Sketch；写好后白天写、晚上跑、第二天我来看。打破不同角色边界，快速交互验证——做内部交付领航、问题处理、最终上线。

内部实践案例三：QuickSilver

第三个是最近刚发布的 QuickSilver。但跟前面有个大区别——我们是在一个存量系统里面。今天大多数真实软件都在存量功能里，可能涉及跨库——前端一个库、后端一个或多个库。

我们的 case：Code IDE 基于 VSCode 开源版本做上层 Agent 能力。底层是非常大的 VSCode 开源代码 + Agent 相关代码；上层 IDE 几乎是 VSCode 开源加我们自己做的一套，主要 TS 写。底层 Agent（Agent Group、知识引擎、跟后端 Infra 通信）通过 Golang 实现。整个仓库已存在好多年——最初服务于灵码、再之前服务于阿里巴巴内部研发提效插件——生命周期非常长，有很多历史难以理解的代码。

第一层是地基层——让 AI 看懂这 400 万行存量代码。结合 Retrieval 能力把代码微细化，让 AI 快速理解，而不是在 400 万行里疯狂 grep（效率非常低）。然后是代码知识图谱。第三个是内部研发流程规范——把历史 Spec 文档落库——落库后 AI 知道仓库之前开发过什么需求；看到不可理解的代码可以找到当时的 Stack、当时的思考、为什么写这段代码——对历史演进过程有比较好的理解。

第二个是杠杆——释放 AI 写代码的能力。AutoStep 能力——不只是 Step，是 Auto Step。 通过多维度、多 Agent 并行做广泛 Design——代码设计或行业知识设计——产出各自 Design 结果，写出相关 Step，做整体合并。Spec 也有 Review 环节（也是 AI 做的）——Review 不同 Agent 写的综合 Spec，做深刻 Review，达到我们叫 Ultra Spec 的效果。最后交付的 Spec 比之前更具体、考虑全面、质量更高。

执行环节通过专家团模式做最大化并行——并行编程、并行验证。执行也需要设计护栏——不断交付海量代码，人没办法做 Review。所以结合刚上线的 Ultra Review——对每个 PR 通过异步并行方式做技术 Review。每个 Agent 有自己的 Review 视角：安全、架构破坏、未来扩展性、代码良好编写、注释完善等。

最后还有一个 Field（处于早期阶段）——结合 Code Use 能力自动复现复测阶段发现的缺陷，复现后修复，修复后录屏验证，最后提出交付 PR。这个过程跟前面 Issue 流程有一定类似度。

整体效果：差不多三周左右完成所有功能开发——这个功能跟最近比较火的 Google DeepMind Five 类似。我们在 400 万行存量代码里新增 50 行代码。 当然今天代码已经不是唯一考量因素——实际 AI 写的代码可能甚至超过 400 万行。我们通过 AI 可以快速并行验证同一个 Stack、同一个想法下不同方案的实现效果，最后通过人或 AI 做最终合并和沉淀。最后绝大多数代码完全由 AI 编写。

新瓶颈——研发流程的失衡

最后我抛一个小问题：工业普及之后，组织真正遇到的可能不完全是提效，而是研发流程的失衡。 原先所有软件企业的人力成本投入大多在 Coding 环节，前面设计、产品演变、后面测试发布占比不同。但今天纯 Coding 效率被提升得非常高之后，有这样几个问题：

  • Review 是个比较大的瓶颈；
  • Agent 更多是在做局部优化；
  • 架构慢慢越来越腐化；
  • 最后导致责任链的缺失。

概要

本次演讲深入探讨了快手作为一家拥有上万名研发人员的企业，如何系统性地推进 AI 在研发领域的应用，以实现效率的飞跃。演讲者王硕分享了快手从早期的"局部 AI 提效"演进到构建全局性、体系化的"AI 工程"研发新范式的战略演进、核心打法与实践经验。演讲强调，在当前技术浪潮下，高效利用 AI 提升研发效能已是关乎企业核心竞争力的"技术性代差"问题。快手通过明确战略、落地实践、打造工具平台和建设知识工程，旨在将 AI 深度融入研发全流程，实现从"在流程中嵌入 AI"到"基于 AI 构建（Build with AI）"的根本性转变，最终驱动从"研发提效"到"组织提效"的组织级变革。

一、研发效能战略与演进路径

战略核心

快手的研发效能战略保持稳定与持续，围绕三个核心展开：

  • 用户价值：研发人员满意度
  • 客户价值：业务交付速度
  • 企业价值：组织效率

演进阶段

经历了数字化、精益化之后，当前已全面进入智能化阶段。

第一阶段：局部 AI 提效（AI First）

  • 倡导工程师优先使用 AI 处理任务，通过在编码、测试等高价值单点环节应用 AI 工具，提升个体效率。
  • 实践发现的瓶颈：
    • 局部提效会被下游环节（如质量保证）的瓶颈所"耗散"；
    • 无法有效转化为整体业务指标的提升。

第二阶段：全局 AI 提效（AI Native / AI 工程）

  • 为突破局部瓶颈，快手在 2025 年下半年开始推广"AI 工程"理念。
  • 目标：将 AI 能力系统性地融入研发全链路，实现跨环节、跨团队的整体效率提升。
  • 核心实践：AI 辅助开发、AI 协同开发等模式，并将 AI 嵌入标准化的研发流程（SOP）。

哪些是变的，哪些是不变的

首先，快手对于研发效能的战略一直是稳定且持续的。这里有三个视角：

  • 用户价值：服务一线同学，让他们的研发满意度提升、日常工作足够高效——更偏个人效率提升；
  • 客户价值：研发效能服务的是业务研发团队、技术一号位。提效要让技术一号位有所感知，让上游产品和业务感知到需求交付在变快——更多关注需求视角，怎么交付更多、更快，做到 time to market；
  • 企业价值：研发效能团队属于基建团队，是公司的一种投资。要向组织、金管委、投资人交代，快手的研发效率和组织效率有明显提升——后面所有环节都会扣到这个战略上。

变化的部分则在战术层面，一直紧跟整个研发提效的逻辑——从一开始做数字化、平台化，到 24 年做精益化（把整个组织效率通过数据的方式管理起来、识别低效环节），到当前阶段，我们认为智能化是快手一定要搞的阶段，而且要在行业里搞到领先。

为什么智能化必须搞——技术性代差

这里有句话希望大家关注：快手认为，如果不够快地做 AI 提效，它其实是一种"技术性代差"。就有点像有的公司已经云化了，但我们还在走传统架构——这个东西未来会造成业务增长的影响。

所以从这个维度来讲，不开展 AI 提效，就等于产生这个代差，会影响快手的业务增长。在这件事情上，公司各个技术一号位、各个业务线 CPO 达成了高度一致——这个方向一定要搞，而且快手应该做到业界领先的水准。

关键打法：从搞实践、搞工具、搞度量开始，但今天要跟 AI 进行有效结合。整个快手的智能化分两个阶段：

  • 第一阶段叫 AI 局部提效，关键词是 AI First；
  • 第二阶段叫 全局提效，关键词是 AI Native。

未来会不会有第三阶段？需要根据当前的问题和认知迭代逐步发展。

局部 AI 提效——理念、实践、工具、度量

如果要实现整个组织的提效，局部环节没搞清楚，全局上就是空中楼阁。所以要先识别高价值的研发活动并对其提效。这个阶段需要在理念上做一些变化：

一开始大家接触 AI 时会担心 AI 对人的替代性。我们要传递清晰的概念——AI First。作为工程师或研发人员，第一步要想我们能不能通过 AI 让它先做，人再去做，引导大家应用 AI，并把 AI 用到关键环节里。

实践层面我们关注个人研发绩效——AI First 是给一线同学听的，所以要关注个人绩效该怎么做、以什么方式开展个人效率的提升。

工具层面提法叫 Build with AI——研发效能工具要把 AI 嵌入进去。整个研发体系非常复杂，不太可能一开始就建一个全新工具改变所有研发习惯。需要在各种研发链路工具中把 AI build deep 进去，让大家在不改变习惯的情况下用上 AI 能力。

第二个工具层面的关键点是知识工程——要把工程师脑中偏隐性的知识足够显性化。如果不显性化，即使最领先的工具也大概率只能做到 60 分，很难达到 80 分。

度量层面，我们建立了 AI 原子能力的成熟度模型，去知道 AI 现在情况怎么样、工具能力怎么样。

软件工程本质：AI 时代的力学平衡

快手 24 年就开始做 AI 提效，到 24 年底发现：好像只用 AI 工具是没有办法提效的。原因是一开始的 AI Coding 解决的是简单场景或模板代码问题，但对涉及工程师深度 know-how 的架构、移动端功耗等设计，AI 没有任何知识。

所以我们回到软件工程的本质思考：以前是人的确定性决策 + 质量测量。AI 来了之后引入了不确定性、概率性问题，就要做更强的质量测量——人力建模要增强、测试环境要增强，反而带来额外的人力投入。

用了 AI 之后产生大量代码，质量保证就要投入更多人力——开发节省的时间会被消耗到质量环节。如果不做质量环节的提效，就无法取得平衡——开发提效会被采样环节的人力投入耗散掉。所以我们去搞智能 CR、搞智能单测，是在这个环节上很重要的逻辑。

第二个逻辑：以前是 AI 工程师跟 IDE 交互、把代码输进去；今天我们要把需求转化成一个非常复杂的 prompt 给 AI 执行。AI 的执行能力已经足够强，但人类编写这套输入非常痛苦。这个认知是在 25 年初做的，后来业内开始提 SDD 方法，跟我们当时的假设基本趋同。

四类研发实践活动

AI 时代要从工具驱动转向实践驱动——实践不局限于工具，让员工充分发挥自主性，把 AI 用到落地。我们抽象了四个实践活动：

  • 标准开发：以前的传统开发模式。其实在标准开发阶段，我们一直都是 SDD 研发模式——写需求分析文档、技术设计文档，本身就是 spec，只是大家不这么叫。在 AI 时代下 Spec、Test、Review 越来越流行，但软件工程的本质一直没变。快手会强化标准开发阶段，把以前的质量护栏标准化保留下来。
  • AI 辅助编码：在测试阶段、发布阶段开始明显出现，可以通过 AI 完成代码编写。
  • AI 辅助开发：产品研发全链路的每个环节都能跟 AI 提效。快手很早就采用了标准化的研发 SOP，会针对每个环节做 AI 提效。
  • AI 协同开发：跟 AI 进行高频互动。

实践层面快手会先告诉一线同学：这样做是对的、是好的——而不是告诉你为什么要用工具，而是你应该怎么样实践、对工作本身进行提效。

工具落地与价值识别

AI 这件事，做效果是挺难的，做工具很容易，做产品也很容易，但把它做成功、让它真的对大家有用是比较困难的。我们对工具的思考是从价值和实现难度上去看——纵轴是价值，横轴是好不好做。通过这种方式能识别什么研发任务是高价值的、哪些部分容易做。先找低垂的果实做，让组织在快速时感知到 AI 起的作用。

整个走向 AI 的理解能力是核心。在设计阶段帮 AI 帮我们读代码（DeWiki、CodeWiki 都能实现）；编写良好的设计 test（不是 unit test 而是 e2e test）；开发阶段做 CR 和端测；测试阶段解决用例、接口用例、应用执行；on-call 阶段解决智能 on-call，减少工程师在 on-call 上的投入。

我们建立了九个工具，按 Local（如 Coding 能力）和 Remote（如 CIT 跟 PR/MR 深度关联）两类划分。快手 25 年底代码生成率达到 30%（其中 AIGC 范式贡献最大），到今天开发者代码生成率已达 50% 以上，仍在快速迭代。CRAG 这块，我们通过 AI Service、线上工程、知识工程对 CR 效果有非常明显的提升。

知识工程的关键

九个工具其实只是一层壳子或一些 system prompt 的构成。要想做得好，很重要的就是知识工程。否则即使是 ABC 常用方法、设计范式，也只能达到 60 分；当我们把知识工程加到 BD、Sail 上之后，有非常明显的提升。

知识工程难在两点：知识怎么来 + 知识怎么低成本维护。

快手把知识分成八类（从右往左看）：

  • 业务知识：业务实体、业务关系、业务规则——来自 PRD、代码、工程师脑子；
  • 研发规范：编码符合快手编码规范——比较关键、需要被记住；
  • 实体关系：不同服务、仓库之间的关系——实现需求时要知道代码该写在哪个仓库；
  • 代码、端上、数据处理能力：市面上的 BP 不完全适用于企业场景，需要大量迭代实现。

知识低成本维护的做法：FAQ 类、仓库型知识从仓库自动抽取；研发规范知识从产品帮助文档、SDK 手册或代码中抽取；右腿的知识（业务相关）通常来自 PRD，甚至像快手做了比较好的私有率管理，可以从测试回归用例里逆向过来。知识做得好，很多产品能做 80 分；做不好只能做 60 分。

AI 原子能力成熟度模型（M1-M4）

  • M1：经过概念验证，场景 OK、价值 OK、复杂度也够了；
  • M2：找到试验团队，能力应用到场景下，全链取得试验团队认可，能达到业务结果——但效果还达不到 80 分，大概 60 分（知识不全、case 理解不够）；
  • M3：成熟期，能力满足大部分场景需求，支撑功能完备，具备大规模推广条件——快手很多能力已达 M3；
  • M4：需要更少人工介入——目前快手单点能力还没有完全达到 M4。

局部提效的瓶颈：传递不到北极星

局部提效取得了一些进展，但有个值得思考的点——人均通过提升不到 6 个点，非常惋惜。我做上面的 AI 局部能力提升，但最终没有传递到北极星，也没有传递到上游业务可感知的维度。局部做得很好，整体却没什么提升，这时该怎么办？

全局 AI 提效——研发范式升级

我们 25 年下半年开始研究、今年在公司全面推广。这件事叫做研发范式的升级——已经不是个体问题，而是组织问题、更广范围的问题。

我们提出 AI 工程理念，强调团队协作，强调产品的构建逻辑会发生变化。从实践、工具、度量三个维度展开：

  • 实践：研发组织级实践场景的落地；
  • 工具：Build and Run——不是在以前流程里局部嵌入能力，而是建立全新的平台。这个平台之上后续建的集约能力都能拉上来，业务也能定制场景特点；
  • 度量：提出需求的 AI 渗透率——以前的目标管理是用户价值（个体阶段已满足），但组织绩效、组织目标层面没有得到满足。

局部到全局的传递问题——三个典型场景

  1. 评估颗粒度没变：以前评一个需求 10 PD，有了 AI 之后还是 10 PD——需求颗粒度没变化，工程师人力也没变化，监控需求不可能有提升；
  2. 团队 AI 渗透不均：还没做到全栈化。前端、移动端对 AI 渗透得很好，后端工程师弱一些——前端在等后端，资源排期匹配不上，交付周期没变化；
  3. 测试环节卡顿：开发环节做了大量 AI 提效，测试环节不够——开发完成了一直卡在测试无法上线，需求分布不变，交付周期没明显变化。

所以协同模式有很多场景需要彻底解决。我们花了很大精力看做得好的团队是怎么集成的、做得不好的团队有什么特点，总结规律。

需求级度量：L1 / L2 / L3

从局部提效（个体和阶段维度）转向以需求为粒度的全局提效——以前编码提效成功率/渗透率很高，但不反映到需求上。我们把所有指标在需求层面反映：渗透率、覆盖率、特征率，都能考虑到需求的个性化。同时建立归因逻辑去识别群体问题。

需求定了三个等级：

  • L1 → L2：快速把复杂系统直接向 L2 升级；
  • L1 → L3 跨越式：比如低代码、自动化能力的需求，直接进 L3——AI 从头做到尾，中间测试简化，整个协作环节全部消除。
  • L2：需要做大量能力赋能 + 团队调整。

团队分类与差异化策略

团队不能一刀切。横轴上分两类：

  • 纯研发团队：结构主要以研发为主，包括做业务的；
  • 产研一体化团队：很多基建团队、平台型团队是产研一体。

需求管路也分级：C1、C2、C3。最下层 C2/C3 是复杂需求，做了架构抽象和对齐适配，全责任型；C1 是稍简单的应用。两个维度切下去，把快手团队切成四大类，每类有相应的提效方法、手段、目标，避免一刀切。

右上角的 AI Native 团队需要重点介绍。它继承前面三个象限的特征和方法，但构建产品的方式发生变化——以前给人用，今天构建给 AI 用。这就是我们提的 Skill First：不是构建一个 APP 给人用，而是构建一个 Skill 给 AI 用，通过 Skill 完成业务目标。

许多原来无法企及的全栈、产研一体化在这个阶段都能实现：产品经理可以写 Skill，没有后端资源依然能完成交付；后端同学以前写接口，今天直接完成闭环业务逻辑。理念和构建方式完全发生了变化。

AI 研发数字化平台

左边是 Local 开发态——可以跟 AI 进行 wrap 互动、对 AI 进行更好的监控；右边是 Remote 节点态——固化简单场景下可以端到端交付，甚至不用打开电脑，在手机上就可以完成需求分析、技术设计、代码交付与测试。快手很多团队通过低代码定制已经能实现 80% 甚至 90% 代码生产力。

整个团队建立在平台化 + 生态化基础上。研发效能生态体系：

  1. 对所有基建进行 CLI 化（仓库、发布系统、CI/CD 流水线）——对 AI 友好；把方法论阶段的研发知识 Knowledge 化；
  2. Skill 技能层：CR 型 Skill、Knowledge Skill、组合型 Skill、专业型 Blue Skill；
  3. Skill Pack 技能包：不同工种、不同同学可以默认下发开箱即用；
  4. Template 智能体模板：通用智能体使用过多时（比如挂 100 个 Skill），性能和能力会断崖式下跌——要采用专用智能体让它更聚焦、有更专注的目标，把 Skill 核心做小、把 system prompt 优化。一方面是技术要求，另一方面是大家的使用习惯——遇到问题喜欢找接口人，专业智能体契合度更高。

需求 AI 渗透率的算法

逻辑很简单：一个需求分了很多阶段、研发活动，每个活动有权重（比如开发阶段 40%、具体设计阶段 10%）。每个阶段定义基线（比如生成率基线 40%）——达到 40% 这一项就合格。每项得分综合得到需求得分。

我们的 L2 是各项得分 0.5 以上，相当于整个研发任务里 AI 渗透率达到 50%。L3 需要达到 80% 或 90%。

这种方式牵引大家：第一关注需求粒度的 AI 应用，而不是单点应用；第二在数据上反映工作流上的传递、协同情况、排期情况。配套归因方法，洞察问题瓶颈点在哪里、做相关改善。

成果

今年年初 L2 占比不到 10%，到今天达到了 45%，目标是年底达到 80%。

人效：原来一个月交 3.3 个需求，目前 3.65 个，半年提升约 10%——而去年全年的局部提效效果还不如这个。从结果看，全局提效才是真正能给组织带来提效的东西。

展望——从 AI 研发提效到 AI 生产力

研发效能在整个组织提效中是非常难的——生产环节、生产链路协同的角色非常多。把研发提效的方法放到人力资源、财务、行政、商圈等其他领域，AI 提效的方法能得到更大的放大。

所以今天，我们已经从原来的 AI 研发提效转变成了 AI 生产力——希望把 AI 作为新型的生产力，带入到整个研发链路当中。

四、CodeWiki 核心策略：从"读完仓库"到"正交拆解"

核心理念：不做一次性整仓阅读，而是拆成足够正交、足够完整的小任务，逐个击破。

  • 每个章节 = 一个独立子智能体启动指令；
  • 子 Agent 内部递归拆图、生成、合并；
  • 关键技巧：
    • 压缩用于探索期；
    • 因果递归用于真正写内容时；
    • 让模型在写作阶段保持在同一表达空间。

  • 使用 rank + batch 两类工具（对应 DeepSeek 在 SWE-Bench 技术报告里"接近 80 分"的实测组合）；

AI Coding 两年范式迁移

这两年 AI Coding 是 AI 赛道速度最快的方向。比较有代表性的是 Sonnet 3.5（其实大家用到的应该是 Sonnet 3.5 New，或者叫 3.6，大概是 24 年 11 到 12 月推出的）之后，AI Coding 智能体发生了突飞猛进的发展。同期还有 MCP、Cursor，也是那个时候大家对 AI 能承接 coding 任务有了初步的体会。模型在编码场景的落地，从代码补全、代码审核也转向了端到端的交付。

现在 SWE-Bench 上的一系列评估已经逐步趋向饱和，包括 Anthropic 前一阵推出的 Lighthouse，几乎把软件工程相关的榜单已经打满了。所以我觉得在通用场景，Anthropic 已经摸到了 API 的边界——当然这是我的理解。随之而来的是程序员需求的变化——从"我需要一个 Copilot"变到了"我需要一个数字员工，需要一个能够帮我把需求端到端真正交付的智能体"。

另外一个趋势：随着模型发展，coding 场景的 hacks 也在不断简化。之前我们一直说 LangChain、AutoGen、LangGraph 等等，最近其实也不再提了。之前我们做一个 agent 大概要上万行代码，现在整体 loop 大概也就是千行级别的代码。 我们的工作重心也变成了——给模型更准确、更丰富多样的上下文，以及更稳定的执行环境，更多的能力拓展。

图中这是我们在小红书真正落地的一个代码仓库——把一个代码仓库变成 Wiki 的展示。它呈现了一份结构清晰、具备架构图、实现流程图、API 文档的代码仓库知识库。它的定位是：把一个代码仓库自动化生成一个结构化、且具备业务逻辑的知识库，帮助人以及 AI 快速上手陌生的代码仓库。

自动生成一个大型代码仓库的 Wiki，本身也是一个非常经典的长程任务。这类任务的难点在于：要持续地把代码仓库文件读下来，持续地撰写文档，整个运行时间是小时级，整个上下文占用基本上是千万级 token（注意这是上下文占用，而不是整个 token 开销）。所以这件事还是有一些技术难度的。

智能体生成代码仓库知识库的前提，是我们能让智能体去理解这个代码仓库。代码仓库理解、代码仓库问答，在 AI Coding 场景是非常基础的能力，也可能是所有做 Coding Agent 的同学第一步要做的——因为它非常直观，也非常好验证。

去年的时候 RAG 还比较盛行，当时我们做代码问答还是以 RAG 作为出发点。但 RAG 现在回看有很多不足：它是零散的召回碎片，只考虑了语义相关性，没有考虑代码结构等等。现在范式也都变成了——给模型工具（list、bash 等），让它逐步去探索整个代码仓库。但这种方式在大型代码仓库中也会遇到问题——比如超过一半代码文件的仓库，模型再去 read、bash、list 的时候，非常容易出现过载。智能体在大场景探索时也容易陷入困惑、难以长期保持目标方向。

四阶段演进：Prompt → Context → Agent → Long-Range

我们把时间线拉回到 24 年。

  • 24 年 · 提示词工程：大家圈选一段代码发到 Chat 界面，告诉模型"帮我解释一下这是什么"。给了模型 20 行代码，模型也只能告诉你这 20 行代码是什么——是一个函数、一个类。但这 20 行代码所携带的语义关系、深层语义，模型看不到。这也是提示词工程很快就触顶的原因——纯粹单次的上下文拼接带来大量幻觉，给模型的东西太少了。
  • 25 年初 · 上下文工程 + Code Review：用 Transformer / AST 帮助模型把缺省函数的上下游调用全部丢到上下文里。这时候模型就可以做 Code Review 了。25 年初其实大家很火热的两件事：代码补全 + 代码审查——都是把对应的 patch 或变更代码贴好上下文给到模型做评估或生成。
  • 25 年中 · Coding Agent 进入视野：以 Claude / Local Code 为代表的 Code Agent 开始进入大家视野，大家开始把 Agent 作为编码主力，开始做上下文工程。在 25 年 4 月到 8 月这四个月，包括 Stack、需求管理等层出不穷。这时单次任务基本上要去到 50K~100K token——做单点需求很简单，但放到代码仓库知识库这种场景，50K~100K 大概只是一个章节。一个中型仓库期望的知识库大概是 50~100 个章节，几十个章节就干掉了一个模型上下文的 1/3 或 1/2。这时就出现了模型上下文能力和需求不匹配的状态。

算一笔账：上下文是被什么吃掉的

给大家一个概念：业务一行代码大概是 15 token，一个完整的 200K 模型上下文大概能吃掉 1 万行代码。除了代码行之外，文件路径是隐形的、容易被忽略的消耗大户——一行 Python 或 Java 的文件路径大概就要去到 30~50 token。最开始的时候，像 Local Code 和 Plugin 这些初级的 Agent 上下文管理做得蛮糟糕的——会直接把目录树展开，很多时候目录树会直接吃掉上下文 2/3 的空间。

看一个小红书的真实场景——也是我做这份工作时一直被老板拷打的：把小红书的安卓代码仓库去生成 Wiki。这一个代码仓库大概是 60 万文件，光文件名就消耗了 180 万 token。

长上下文不是全精度上下文

这周的时候，Gemini 的一个核心开发者 Andrew Guy 来大陆了，我和他交流了一下。Gemini 的 1M 上下文是最早推出的，但他认为现在的 1M 上下文模型还不是全精度的上下文。原因有两方面：

  1. 1M 框架现在还没有一个比较好的评估手段——所有 1M 模型对框架评估还都是通过 MRCR（大海捞针，随机插点看模型能不能召回）；
  2. 1M 框架在训练时也都是以搜索任务为主，让搜索的候选把上下文撑得很大，靠搜索的结果准确性给模型奖励反馈。

带来的问题是：这样训练出来的模型，只能保证 1M 上下文后半部分精度很高——如果你的召回物料在中间部分（比如 500K 或 600K 的位置），模型精度也很差。

典型例子：四月份 DeepSeek V4 发布，给到了 NSA + MHA 的 KV-Cache 压缩技术。我第一时间把它拿来用——一开始想，既然给了非常低成本的 1M 上下文，那能不能干掉传统搜索任务，把所有候选都直接丢到 1M 上下文里、每次直接追加问题？试下来不是很行。 所有的 1M 模型现在都没办法保证全精度——精度最高的是 Opus 4.7，但它的 1M 上下文精度也下降了。Andrew 给到的说法是：4.7 的 MoE 激活专家变少了，是为了把成本降下来。

这是一道适合作为团队准入门槛的考题

这个东西非常适合作为一个团队的阶段性基础考题：

  • 非常能明确验证微调生成对不对、接口文档写得准不准——这些一眼就能评出来；
  • 它是典型的长程任务——要面对的是上万的代码文件、几千万 token，把它消化成几百章节的页面，整体也是 10 亿量级的任务；
  • 任务要素足够丰富——你需要多智能体、需要非常精炼的计算，需要让 Agent 能够保持稳定的长时间运行。

解法：从"读完仓库"到"正交拆解"

面对这张卷子，我给到的答案就是：不去一次性把整个仓库读完，而是把它拆成足够正交、足够完整的小任务，然后一点点做。 围绕这个答案提了很多方案：两阶段任务、各种压缩策略，以及代码仓库的划分策略。

挑战目标：把一个 60 万代码行的小红书安卓仓库，让它准确且完整地生成 Wiki。模型本身只有 200K 的代码库容量，要想办法让 200K 的模型把这 60 万文件干掉。

采用分片的方案去拆开，把它变成 N 个 200K 的独立小任务。

  • 第一阶段 · 大纲生成：让大纲 Agent 用粗的阅读不断拆解，把代码仓库拆成很多小份，生成一个大纲，再基于大纲中的章节启动智能体专门生成对应内容。大纲生成阶段的目标是让它具备战术师的视角——尽量看更多、看更远，不下钻细节，通过逐步、逐层展开形成全局理解。
  • 第二阶段 · 内容生成：大纲生成智能体输出章节任务，每一个章节任务可以看做一个子智能体的启动指令。子 Agent 根据这个指令展开，不断递归把任务拆成子张图，所有子张图大图生成完后再交给生成模型逐步成稿。

工程化细节

在子目录管理上，我们留了 50 片的部分，每片大概 5000 个文件——但安卓仓库有 60 万文件，怎么办？通过对表层重复不下探到深层，逐层的展开能带来更好的全局理解。

关于命名的一个点：判断项 / 辩论项不能用 "Home dash" 这种直接被模型捕获的代词——所有的判断项、辩论项通通都要用代词去抽象。这其实是 DeepSeek 在 V3.2 的技术报告里提到的，他们使用了 rank 和 batch 两个工具，让 SWE-Bench 接近 80 分的成绩。所以大家就着模型技术报告能看出 Coding 模型能力的提升路径。

整个仓库的拆解分析能节省大概 90% 的成本，只保留足够的牵引。我们把模型拆成一个一个相对独立、相对正交的模块。

另外，我们参考了一些唯一在互联网公开的实践——他们用到了地理相关信息：相关请求的相关门店大概率会出现在同一地理区域。所以同样的思想可以用在代码仓库的子节点划分上——第一层节点大家是同样的，但到第二、三层时需要用不同的子节点交代任务，让每个子节点都有清晰的任务边界。

真正写内容的时候，要让模型保持在因果递进的同一表达空间，而探索期则尽量上下文压缩——这两阶段的策略不一样。

开场与议程

今天我们基于 AMC 算法的方式来解决 B/C 链路上的变更超发问题。我会先介绍我们目前在广告 B/C 链路上的问题和痛点，再讲我们怎么解决这些问题、系统是怎么设计的、过程中遇到了什么问题、怎么解决的，最后做一些实际效果和经验分享。

先达成一个共识——这里说的 B/C 链路主要指广告场景下的 B/C。

三层架构与变更痛点

整个技术架构大概分为三层：

  • B 端 · 广告投放端：支持广告主在平台上完成一系列广告投放意愿的表达——投放在哪个人群、哪些地域、什么时间段、什么样的价格。
  • C 端 · 客户端：在客户端发起广告账户经营，需要把广告主的一系列决策、投放诉求传输到 C 端链路。
  • 数据管道：相当于数据搬运工，把数据从 B 端搬到 C 端。涉及实时同步任务、批量传输任务，以及 B 端广告主表达数据怎么映射到 C 端状态。

每一次变更——比如新加一个广告投放策略——会涉及字段的修改或新增。修改字段时它要改自己的扩展结构、改字段模型，对应输出可能是 N 个服务的变更。变更完之后，中间数据传输也需要适配，再到在线层用到的字段需要和 B 端字段的含义值映射。

当前变更周期大概是这样：我加一个字段，平均上线之前需要三个团队、走 10 多步——口头约定、各自回去改、改完通知、相应验证环节、上排档、上配套等，每个阶段验什么也是大家在群里口头沟通。每次至少三个团队，可能更多——类似于三件式架构问题，包括一些算法模型问题，团队数量更多，人工操作步骤更复杂。

四大痛点

  1. 人传人手工约束：所有系统变更通过人工手手相传——开始约定字段是大写，到中间调用层用的是小写开头，再到机关库时按小写或字段顺序使用——整个过程会存在失真。
  2. 影响面靠人的经验判断：比如把字段从 int8 扩成 int64 做扩容——是不是能保证整个链路上所有数据、所有服务对这个字段的容量都做了扩容？这完全靠人经验。
  3. 追溯断裂：互联网迭代快，字段变更周期长——一段时间后再去往前追溯、为什么改这个字段，背景可能就没了，或者有经验的人也走了。
  4. 验证零散：缺少统一记录的地方。

目标态：一句话驱动全链路

我们整体希望达到的状态是：研发通过自然语言一句话表达诉求，把刚才提到的人传人口头约束、各自验证的方式，自动化串起来。整个过程通过一个 Agent 完成。

研发在 IM 群里和 Agent 通过群机器人对话，把需求理解出来，然后基于变更需求做小编排和调度。配合质量评估能力来完成中间调度。中间会有人工卡控——因为变更是严肃场景，不可能所有动作完全让 AI 自主完成。在严重变更场景（比如上线、上库），引入了人工审核的机制。每个环节执行完做对应验收，最后形成一个含背景、对话、验证结果的完整知识链报告。

产品架构 · 主从双层 Agent

采用 Spring Security 风格的身份协作模式做架构设计。

  • 主引擎 Actor Engine：负责编排、判断影响面升级、推进流程，作为监督组和指挥官来操作。
  • 业务域子 Agent：实际执行者。为什么跨业务域？因为变更涉及跨多个业务域。

这部分设计的目标是解决整个代码的重复出现问题——通过专业词典来准确表达。

标注分两类：

  • 意图理解、影响分析——作为 Action 内部的核心能力；
  • 独立于当前字段变更 Action 领域外的辅助能力——通过对接的方式接入。

记忆与规则体系

每个场景的变更都有一个产生要求规范——类似一些红线动作，以及不同场景下的行为指导和用户引导。基于这个还有一部分是变更过程的文档内容：

  • 环境变量：消息是给谁发的、对应的角色（前端、后端研发、QA 等）能不能做这个事情；
  • 当前流程信息：当前变更的基础信息、群里需求沟通记录、整个变更流程；
  • 历史记录：每一步执行的记录——这个字段在哪里加了什么；
  • 应用规则：核心动作变更（比如端口发布到对象，必须有技术负责人确认；某个服务在变更路径里已经做了变更，要求回滚等高危操作必须二次确认）；
  • 长期记忆：把历史变更配置、怎么变更、验证点在哪里作为 Action 知识，给到 Action 层完成编排动作。

底层框架 AKM

这一套是我们厂内的开源 Java 框架（讲者称 aikm），定位是企业生产级架构。选择它的原因：

  • Java 实现，与厂内系统有天然适配；
  • 厂内已经有很多落地场景；
  • 原生集成了厂内一些可观测、变更平台等基础设施。

挑战 1 · 字段血缘自动维护

变更一个字段需要知道这个字段在哪些环节使用——这就需要完整的检验链路。靠人工经验梳理是没问题，但字段一直在变、使用方式也在变——怎么保证字段血缘一直自动更新？

我们引入了白盒分析能力——对中间数据管道这一层做重点分析（它本身天然具有映射属性）。具体做法：通过代码仓库扫描和理解 + 阿尔法分布式配置 + 投放端库表结构 + 在线索引层的使用，把这些关联起来。这样可以拿到一个字段在 B 端对应的字段信息、在 C 端哪些服务上用了，得到完整链路。

基于字段血缘 + 字段变更意图，自动推出影响哪些表、哪些服务、各自 Owner 是谁的影响评估报告。基于这个还可以做变更拦截——如果影响面对应的 Owner 没有确认，变更直接拦截；S0 服务必须经过哪些阶段才能动。

挑战 2 · 人员流动与历史追溯

过去一段时间后再回头修改、问当初为什么改，可能找不到当事人了。我们的解法：群里所有人记录都作为长期记忆存下来，存到计划存储里——包括各个 Action 的执行详情、聊天背景、确认澄清过程，整个过程留痕可查。

另外是工单的追溯：本次交互过程会按照需求工单方式系统性管理；内部变更平台对应的执行流程（部署了哪些集群、什么时间段变更、谁部署确认）也统一到这个变更过程里留痕。三端变更各阶段的验收（测试、谁确认、怎么验收）也做流动报告。

最终把这些信息流转到知识规则里，让变更流程越用越聪明。

SCP 协议与业务域拆分

为什么要拆业务域？投放端变更涉及数据库变更 + 分布式服务；中间数据管道涉及大数据任务、云管配置、代码修改。如果把所有变更能力集中在一个 Agent 上有两个问题：

  1. A 系统变更率比较高，迭代和运维属于不同团队，相互影响；
  2. tail 问题——所有信息全给一个 Agent，但并不是每个业务场景都需要所有字段信息。

所以由主引擎 Actor Engine 负责整体主控，业务域 Agent 作为子 Engine，通过 SCP 协议和主引擎交互。Agent 能力依赖通过 Skill 方式注册到 Actor Engine 里。

最终交互形态 + 推广效果

研发不再需要跑工单平台、变更平台、数据任务变更平台，只需要在群里和 Agent 交互——通过发送指令完成变更。

变更周期从原来的 3~5 天，现在两三个小时就可以完成，最快 1 小时起。变更遗漏和顺序错乱的问题降到零，整个链路 100% 可追溯。每天 token 成本大概 10 余块钱（用千问类模型）。

三个经验

  1. 核心动作不允许跳过：研发觉得流程麻烦也不允许，强规则保证。
  2. 人工审核绝对必要：在每个核心变更场景之前都引入人工审核动作。不允许"为了把流程跑得更快而让模型自由发挥"。
  3. 异常处理：每一步要允许重试 / 回退；异常要及时上报、人工介入；知识积累让 Agent 越做越强。

Q · SCP 协议在 Skill 与 Action 间怎么传 response？

SCP 协议通信时先创建会话拿 Session ID，再根据 Session ID 把要执行的事情通过协议推送过去。同步任务通过流回传；持久任务（比如分级发布）下发工单后定时轮询工单状态。Skill 只定义能力约束（你能做什么），不定义实现——主引擎按需调用。

Q · 工作流任务超时 / 审批超时怎么管理？

底层是传统服务 + 状态机——Actor Engine 的底层就是一个传统服务，每个用户开始交互时都创建一个虚拟变更需求单。Workflow 中 Plan 超时、某一步执行失败、验收不通过等状态都统一管理到需求工单里。需要重新拉起或回到某个节点重新流转都告诉 Agent，每一步都可重试 / 回退到指定节点。

未来规划

未来希望各端 Agent 之间不再通过 SCP 协议交互，而是直接在群里通过自然语言流式协作；最终目标是把数据变更 Agent 做成小红书全公司的数据变更 Agent 能力。

压测为什么开始失效

"失效"不是说压测没法压，而是压测结果不具备性能分析、优化指导的意义。这背后和大模型的负载特征有关。

传统压测 vs 模型压测

传统压测数据结构相对固定——比如小红书首页就是传 cursor、uid；视频笔记就传 note_id，走推荐引擎链路 + 中台链路。参数组合可枚举（发文本笔记、视频笔记、传图片）。我们用 QPS 表达容量特征，用 RT 和错误率表示负载边界。

模型压测不一样：

  • 输入结构不可枚举——用户用一句自然语言表达，模型就开始进行一系列分析和输出，没有可参考的枚举值；
  • 相同 QPS 下，所有 input/output 输出对模型的压力不同——长链路、长上下文、多轮对话的 batch 是不确定的，整体过程不确定；
  • 模型压测更关注 TPS、吞吐和 GPU 利用率——和 QPS 这种指标不同。

结论：从传统压测的"流量约等于请求数量"，到模型压测的"流量更趋近于分布和内容的差分"。

为什么聚焦数据构造

大规模压测的可信度，首先取决于数据构造能否真正还原负载。压测分几部分：

  • 压测平台构造——大多数公司现在已经有千万级 TPS 发压能力，发压能力不是瓶颈；
  • 压测流量的编排——每轮先发什么数据后发什么数据，重要但不是最突出；
  • 数据质量——直接决定压测质量（输入长度、输出规模、上下文层次都包含在内）。

我们容易构造模板化、均值化的数据——压测能跑起来，但很难暴露真实业务的复杂场景。压测跑起来但没有指导价值。

压测看起来没问题，上线为什么暴露问题？

压测构造的结果是 RP 平稳、错误率很低、利用率正常。但上线后 TFD 存在很大波动，长尾 RT 长时间跑场景测试甚至产生 OOM，吞吐也偏。

偏差在于：压测和线上的短 prompt 及长上下文对比、压测单轮和线上多轮对比、压测简单任务和线上复杂任务对比。

数据构造的"四难"

  • 真实性难还原（"像不像"）——线上分布不均，线下压测过于均匀化、同质化；
  • 覆盖度难保障（"全不全"）——压测覆盖的用例是否全？是否只包含单轮问答？只有简单语义？
  • 复杂度难分层（"分不分"）——复杂度有没有分层？有没有设计高复杂度场景？
  • 有效性难判断（"行不行"）——结果是不是置信？对调优有没有指导意义？

详细看：

  1. 真实性：线上流量不是真实分布——有长尾、输出规模差异大、上下文层次不一致、任务类型占比不同；压测样本却很整齐，长度集中、场景单一，缺少复杂性。这会低估 TST 和长尾 ID，高估存储能力，容量边界判断失真。
  2. 覆盖度：以"点点 0.3.0"业务为例——它包含视频卡片、图文笔记卡片、POI 个人页等多种卡片，包含攻略模式、普通论坛模式、任务模式等多种业务特性。刚接版本压测时只覆盖主链路（拉新任务的主页问答 + 文本打例），主页打例不管线上是否有人看过；低频高成本场景（POI 第三方查门类、经纬度信息）和复杂检索工具都没涉及——复杂度瓶颈没暴露，外部依赖耗时也没体现，真实容量风险被低估。
  3. 复杂度难分层：业务相对复杂，如果不分层，监控样本会融合到一起。压出问题之后不可解释——不知道是哪一块业务带来的容量边界。每个上下文类型带来的容量边界、长尾在某段时间变长，这些都覆盖不到。
  4. 有效性：哪怕做到了真实性、覆盖度、复杂度处理，也很难判断数据是否有效——风险没暴露，容量判断乐观，优化效果无法度量。

核心转变：从构造请求到负载建模

这四个难点的共同点是——过去更多在构造请求，现在要慢慢转向负载建模。

案例：点点 0.3.0 的两次迭代

故事开始：去年我们接手 0.3.0 业务，需要保证它的性能。

这个业务可以针对社区内容进行检索、提问，相比 2.0 增加了多轮上下文、社区卡片交互和工具模式——同一次请求 token 消耗 + 系统压力和之前完全不同。

第一版：传统流量录制

从工程上录制流量，做敏感信息清洗脱敏（用户 ID、session 等），直接作为压测数据发下去。同时为了保证覆盖性，按多轮 / 文本 / 图片 / 卡片模式拆解构造数据，按场景准备物料（笔记、商品、POI 评论等）。

关于物料：选择用户阅读过的笔记——通过视频笔记、底详接口拿物参的 UID 拼接。这里有个选择——可以直接从 Hive 表笔记后台拿用户所有笔记信息，但这个方式后来被舍弃了，因为它不真实——他今天有的笔记不代表他在这个时候用。

压出来的结果：首轮推理输入长度压测是 20K，但放量之后线上是 12K；推理 TTFB 压测 3 秒，线上是 500ms。卡片占比方面差异更大。

问题分析

讨论后发现：场景补齐了，但负载结构还不够——压测数据的卡片占比 25% 但线上是 20%，卡片线宽部线上用得更多——整体结构负载变重，相对形态偏离了线下真实形态。

第一版语料构造没有看线上语料分布。

第二版：引入线上语料分布模块

加了一个模块——把线上语料分布提取出来，分析关键词，根据关键词生成目标配比。这样更贴合线上。

反直觉的现象

第二轮压测之后发现：流量上涨，但 TTFP 和首发包反而呈下降趋势——这很奇怪，正常流量上涨 TTFP 和首发包应该是商业过程。

分析原因：推广新增的用户主要是新用户，他们大量流量来自萨克斯（评论引导）这样的场景。这种请求上下文更短，新用户没有历史名字，token 消耗更低——对整体性能负载更低。

当时讨论：要不要根据新用户场景再压一次？结论是需要、但不重要——证明当前压测负载能力符合现在预期，但这并不是我们对未来的预期状态。我们会引导用户慢慢贴合预期使用方式。

压测要不要完全贴合线上？

不应该完全贴合线上。 用户使用深入、活动叠导、日常功能迭代之后，过度拟合线上的用户行为会低估未来的负载风险。

所以压测要单独设计超长输入、极端多轮、工具调用，与常规的线上场景压测并行进行，提供多重负载架构保证链路稳定。

有效性校验四件套

数据是不是真的可信？四个维度：

  • 分布校验：token 场景、长尾比例和放量后线上是否一致；
  • 数据质量：过滤无效 query、空上下文、异常样本；
  • 链路校验：所有 throughput 节点都能正常触发；
  • 回归校验：优化前后固定数据卡片，保证结果可比。

这四个点都在解释同一个问题——压测结果到底值不值得信。

工程化落地

需要回答两个问题：数据从哪来？数据往哪去？

数据从哪来 · 混合构造
  • 线上流量复制（工程侧或 Agent 侧录制真实流量）——保障真实贴合，缺陷是出问题不知道是什么内容导致，数据没被拆分，低频场景遇不到；
  • 手动构造——枚举用户场景，通过模板拼接 query，从其他数源拿素材（比如线上捞媒介 ID、个人页信息）；
  • 采取混合构造——同时录制线上流量和手动构造流量。
数据往哪去 · 分场景建模
  • 全量仿真复刻线上流量——接近真实业务权重，但成本高、结论难解释；
  • 分场景建模——每个场景单独建模、单独成能力。压测时按权重发压，线上请求权重变化时只需调整权重，不需要重做数据。

四个核心结论

  1. 数据决定了压测是否可信；
  2. 压测数据构造的本质是负载建模，不是请求构造；
  3. 高质量数据要同时关注真实性、覆盖度、复杂度；
  4. 工程化沉淀比一次性造数更重要。

一、端到端测试的痛点

  • 三大特征：长域 / 长链路 / 组合爆炸；
  • 电商场景 5~6 个领域、新人手工造数 1~2 天、组合数指数级；
  • 传统解法（原子化封装 + 流程编排）后期腐化——底层封装仍依赖人，业务一抖动整套就要重写。

二、测试执行 Agent 的目标能力

  • 动态编排：给一句自然语言诉求 / 用例文本，自动推导出执行计划；
  • 韧性执行：对接口变化、参数错误、异常返回有 retry / 自我修正能力。

三、整体架构

  • 接入层：IDE / CLI / 测试同学网页（多入口）；
  • 规划 + 执行层：测试执行 Agent（Plan → Q → 执行）+ Coding-Agent 框架；
  • 能力底座：业务知识库、接口列表、跨环境网关、脚本管理（Git）。

四、关键差异：测试 Agent ≠ Coding Agent

  • Coding Agent 通用能力强，但不懂业务；
  • 测试 Agent 必须做两件事：看懂业务 + 写好脚本。

五、看懂业务：动作—实体—属性三元拆解 + 链式倒推

  • 把测试诉求拆成
    • Action（下单 / 发货 / 结算）
    • Entity（订单 / 合约 / 结算单）
    • Attribute（实体上的状态 / 值约束）
  • 从"右端"（最终交易域）倒推：要验证买手对账金额 → 需要结算单 → 需要订单 → 需要合约 → 需要商品 / 用户 / 地址；
  • 意图模糊时主动澄清，避免大模型自由发挥。

六、知识库策略：故意不选 RAG

  • 测试领域召回不全 = 漏测；
  • RAG 适合海量信息相关性召回，但对召回完整性要求极高的测试 Plan 来说不合适；
  • 采用"按领域 / 按工具的按需加载"——逐层拉 Schema，避免大模型瞎猜。

七、写脚本：Debug-First + 流量回放

  • 先用向量召回历史脚本；命中即复用；
  • 不命中走 Debug-First：命令行逐个调接口，探索 input/output Schema，再生成脚本；
  • 用流量回放验证生成脚本——把脚本作为被测对象，用前序 Debug 阶段的 input/output 流量做回归；
  • 沉淀两类经验：
    • 工具级 tip；
    • 成功路径脚本作为下次召回的参考样本。
  • 10~20 个用例后新用例一次成功率显著上升。

八、Prompt 实践：不是技巧，是结构

  • Prompt 的本质是把团队对业务的"规范化描述"喂给模型；
  • "问题拆得足够小、每步都有具体产出物"是稳态执行的关键；
  • 模型按填表方式输出，比让它自由发挥要稳定得多；
  • Agent 的另一关键是"记忆"——知识库 + 脚本经验双库联用。

九、落地场景：数据构造

  • 切入点：下游业务的造数 = 上游业务的自动化用例；
  • 当前在 10 多个电商类产品上应用；
  • 未命中缓存：分钟级完成造数；命中缓存：秒级；
  • 单次成本 ≈ ¥1.5 / $0.20；优化后预计可再降 20%+；
  • 真实演示：自动拆 Action / Entity → 调度账号 / 地址 / 下单 / 支付链路 → 命中入参错误时自我修正 → 输出完整结果。

开场

第三点的话，在于我们怎么让 AI 模型学会去做测试——这个东西主要涉及到我们刚才的功能体验和这种自动化效果，以及未来的发展方向。

先以这个图作为开场——其实在传统的自动化模型里，大家都会用经典的金字塔模型。但随着实际业务的发展，目前大部分情况下都已经进化到蓝海形态——集成测试 / 接口测试比重越来越多，端到端测试自动化也多起来，反倒是单元测试慢慢变少了。

这是一种业务测试策略的选择——快速迭代或业务逐渐变复杂的情况下。但同时也反映一个问题——为什么端到端测试依然没有变大、没有演化成高性价比的模型？核心逻辑在于端到端测试自动化成本还是比较高——受限于成本，所以没有变化成金字塔型。

所以核心问题是：怎么用 AI、用 NLP 技术解决这个问题，降低端到端测试成本。

端到端测试三大特点

  1. 长域——比如电商场景，社交、商品交易、支付、物流、售后、履约、购物车等一系列领域，每个领域有不同同学或微服务支持，整个链路下来要找 5~6 个同学沟通。
  2. 长链路——每个领域都有不同功能。一个新同学手工造数自己第一遍走下来，可能就要一两天。
  3. 组合爆炸——每个场景、链路都有很多场景组合，是经典的测试组合爆炸场景。比较典型的表现是手工造数小时级，然后自动化覆盖：现在大多数都在合规场景或主场景，边界 case、异常 case、小场景 case 基本上很难覆盖——因为成本太高不划算。

简单例子：最开始普通订单，找一个 C 端账号、商家账号，构造商品、加购物车下单结算支付——一个链路检查也就 6 分钟。但比如做"买手"形态——买手要做营销、促销，过商品时要配置（哪些地区、做哪些买手机、搭配什么买手合约），结算或分润时基于这些合约做相关逻辑。整体业务场景、领域深度都得到极大变化——结合到具体的买手输出（买手状态、合约生效时间、分润比例、带货方式、流量搭配），整体测试场景就变得异常之多。

过去的工程化方案为什么腐化

几年前大家有工程化方案：原子化、封装 + 流程编排。原子化封装是对服务端接口做包装（工具引入、自定义代码、HPC 请求都属于这类）；加上流程编排（写代码编排、DSL 配置、可视化拖拽）配合参数 run 用例。

这种方法初期大家用得很爽，能解决跨域成本高、长链路编写、组合方案的问题——原子化服务可共享，拖拽编排节省人力。但后期随着业务爆炸发展，逐渐觉得没那么 work 了。

核心点：原子的结构封装还是依赖于人。业务发展越来越快，经常一个接口加几个参数、改几个参数、扩充什么东西——原来的接口就要重调；流程上比如加个买手域，很多配合就比较麻烦。本质是基础能力还是人在维护，会出现腐化，特别是测试架构设计上。

根本矛盾

选的功能化工程化方案——用人工预置的静态编排预设逻辑，想去解决不断上升和动态变化的业务问题。两者效率不匹配，没法跟随端到端测试的快速更新和落地。

所以 AI 时代我们想用 AI Testing 思路——不要人去写原子化结构、不要人写动作的流程封装，全部改成 AI Testing 自动执行。

两种技术选型

当时有两种选择：

  1. 把现在的流程编排方案做智能化——比如把已封装的原子化能力转成 MCP，让 AI 自动调、自动编排；
  2. 更直接——推翻原来逻辑，让 AI 直接感知原生的服务端接口，不做封装让它自己去理解。

我们选择了第二种——虽然对之前的基建没有什么帮助，但它完全脱离了人，上限更高，对后续持续迭代和发展有更好的帮助。我们定义了一个测试执行 Agent。

测试执行 Agent 两大能力

  1. 动态编排——告别人工预设的流程编排，根据用户诉求（造数需求 / 用例诉求）自动推导：先找 A 接口、再找 B 接口、再找 C 接口。
  2. 韧性执行——接口有变化、参数错误、异常返回时，能 retry、自我修正。

整体流程

左边是简单输入（用户自然语言、文本用例都可以）；进入知识推导过程（背景知识、业务知识需要参考）；推导测试执行计划（to do list）；做决策——和存量用例脚本做召回：

  • 能召回原有脚本就直接执行；
  • 没召回就生成——拿到一堆接口先调一把、再写脚本、试跑、有问题就调整；

最终产生测试报告，比较好的脚本沉淀下来作为后续参考。

测试 Agent ≠ Coding Agent

核心区别：Coding Agent 具备极强通用能力，但不了解业务——业务概念不清晰（比如"纠纷单"和"平台介入"语义上是同一件事它不知道）、业务链路（上游是支付系统、结算系统、清付系统）不知道、接口规范（input/output schema、接口优先级和关联）不清楚。

所以测试 Agent 必须做两件事：看懂业务 + 写好脚本。

看懂业务 · 三元拆解 + 链式倒推

用户给的复杂意图，给到大模型可能会丢一些东西、或自己增一些东西——这对测试领域是致命的，因为测试需要增强确定性。

我们让大模型按 Action / Entity / Attribute（动作 / 实体 / 属性）三元关系做拆解：

  • Action：下单、发货、售后、结算；
  • Entity：买手分账订单、结算单、商品；
  • Attribute：实体上属性需要达到什么状态或具体值。

这一步也可以对冲突、缺失的地方做需求澄清。

从实体和 Action 做原子化触发——从最右边交易域开始往前推：要验证买手对账金额 → 一定有结算单（对应支付结算域）→ 结算单需要订单（交易域）→ 结算时参考买手合约（买手域）→ 推到最原始的测试商品 / 买家账号 / 拍摄账号 / 商家。

实际跑出来的图：购买显卡商品并下单支付——五个步骤：创建借卡商品 → 获取用户账号 → 获取收货地址 → 下单 → 支付。

知识库设计 · 故意不用 RAG

业务知识库分两类：纯粹的 Knowledge（业务知识文档）+ 工具列表。两级目录树结构，对应组织 / 业务关系。业务内容样例不用写特别详细，主要是核心概念、领域能力介绍、核心流程介绍。工具列表直接从 Git 仓库或者 Code API 同步过来。

知识库喂给测试 Agent 时没有选 RAG 方案。原因——RAG 适用于海量数据下的相关召回，但测试计划执行过程中召回的全面性要求非常高（漏召回 = 测试计划有偏差）。所以我们牺牲了一部分 token，用按需加载的加表方案——按一级域、二级域、工具列表逐层加载具体 schema。

例子：先加载交易域 → 识别到前置依赖买手域 → 加载买手域 → 发现前置依赖商品域 → 加载商品域。这样能拒绝大模型瞎猜——大模型本身有电商基础知识，但每个公司的电商模式有差异，它有时会瞎猜。

写脚本 · Debug-First + 流量回放

有了 to do list 后：

  1. 向量召回历史脚本——召回不出来也没关系，因为可以动态生成；这一步只是为了加速 / 经验参考，对准召率有容忍度。
  2. 没召回就编写脚本——采用 Debug-First 策略：通过命令行调入各接口，探索接口的各种出参 / 入参，一步步把它放起来；基于调试完成的记录最终生成脚本。一次基本可以达到一次成型。
  3. 流量回放验证——把测试脚本作为被测对象，用前序 Debug 阶段的 input/output 流量做回归。

为什么 Debug-First？传统方案是写比较表 + 单测报报错 + 加日志，循环几次。Debug-First 这种形态一次成功率大幅提升。核心差异在于：实际业务接口的 input/output schema 没办法让大模型一眼看懂、直接生成 code——毕竟接口是直接生案的业务接口，不是大模型理解的接口。

经验沉淀两个方面

  1. 工具积累的调试经验——调试过程中遇到的坑沉淀为工具的 tip。比如下单接口返回 OrderID（前面是版本号 + 分隔符 + 包裹），支付时需要按这个逻辑把 P 删掉、按后续映射过去——大多数模型给两个入参表现形式 + 实际值能推导，但没给实际值它会直接把 OID put 进去。把这种经验沉淀下来，下一次执行就不会犯这种错误。
  2. 脚本成功路径作为参考样本——每次都做脚本召回（用执行计划召回）：完全匹配就直接执行；不完全匹配就让大模型参考原有脚本生成新脚本——一方面保持流程、接口使用形式、业务规范一致；另一方面参考度高，成功率不会再犯错。

沉淀 10~20 个用例之后，新增加用例的一次成功率会非常高。

Prompt 实践 · 不是技巧而是结构

两个核心思考：

  1. Prompt 是 Skill 形式呈现——能串联整个测试用例计划生成的推导逻辑。Prompt 有很多技巧（COT、Few Shot 等），但最本质的是对业务领域的深入认知——比如刚才讲的三元拆分、链式依赖倒推，这是规范化的描述。要把这些喂给大模型，它才会按这个逻辑产生比较好的结果——大模型是概率性模型，简单一句话给它会有各种各样的想法。
  2. 问题拆得足够小——交互驱动：每一步都要求具体的产出物，对产出物做约束。填表逻辑的准确性和稳定性比让它自由发挥要稳定得多。

另一个大认知：Agent 需要记忆——这里提供两个东西：知识库 + 脚本经验。这两方面对大模型的横向迁移能力 + 自我进化能力都提供了好的效果。

落地场景 · 数据构造

切入点：下游业务的数据构造 = 上游业务的自动化用例。所以先从造数切入——某个领域自动化造数解决了，就可以推导到后面的运营执行过程。

当前在 10 多个电商类产品上应用：

  • 未命中高频场景缓存：分钟级造数；
  • 命中缓存：秒级；
  • 成本：单次约 CNY 1.5 / 0.2 美元——这个还是比较早的，近期还可以再降；很多优化措施还没做，预计可以再降 20%+。

真实演示

一句话需求："beta 环境购物车购买显卡，手机号 1234..."

第一步：查造数脚本缓存——没命中走动态造数。Action 是创建一笔订单，Entity 是订单。环境 beta，商品类型借卡，操作方式购物车购买，用手机号查询。

开始推导，过程中产生表格判定：工具是否存在、用户账号获取是否 OK、购物车流程是否能匹配下单支付流程、参数能力对齐——比如有些用户需求"构造一个带备注的订单"，假设没有这样的接口，"下单支付"这步就会显示差。

最终拿到执行图，一步步执行。这个例子里出现了一个问题——获取用户脚本第一次入参有点问题，基于动态返回（参数校验不合法），他重新调整完成。最终形成造数结果（订单信息、商品信息、收货地址）。

命中脚本的情况

比如发一个视频笔记——他先去造数库里 search 召回，召回完后做一系列评估（点击率有 6000+），直接复用。
